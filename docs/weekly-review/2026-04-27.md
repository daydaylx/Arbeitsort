### Positive Observations

- **Room schema export enabled** (`AppDatabase.kt`): Switching from `exportSchema = false` to `true` unblocks proper `MigrationTestHelper` usage and is a prerequisite for long-term migration safety. Good call.
- **`WorkEntryDerivedStateNormalizer` is well-designed**: Injectable `now: Long = System.currentTimeMillis()` default parameter makes the normalizer fully deterministic in tests without requiring a test clock abstraction.
- **Deadlock prevention in `withDateLocks`**: Sorting dates before acquiring nested locks is the correct algorithm for preventing deadlock. The implementation correctly uses `distinct().sorted()`.
- **`offDays` semantic correction** (`AggregateWorkStats.kt`): Recomputing `offDays = freeDaysWithTravel + freeDaysWithoutTravel` (instead of subtracting `visibleWorkDays` from total) is a meaningful semantic improvement that aligns the metric with the classification layer.
- **`ExportPreviewViewModel` bounds guard**: The `MAX_ENTRIES_PER_PDF` check before calling the exporter is a clean, user-facing defensive improvement that surfaces the error early with a localized message.
- **Migration 15→16 now covers SCHULUNG/LEHRGANG** (`AppDatabase.kt`): The bug fix that extended the `shouldConfirm` / `isWorkLike` logic to include the two additional work-like day types closes a correctness gap that would have left those entries in an inconsistent state after migration.
- **`PdfExporter` meal allowance column** now shows the euro amount instead of a simple breakfast boolean, giving users actionable information in the export.

---

### Top 3 Recommendations

1. **Bound the `writeMutexes` map** (`RoomWorkEntryRepository`). The current unbounded `ConcurrentHashMap` is a slow memory leak in a `@Singleton` service. Implement a simple bounded cache or evict entries after each lock release. This is the highest-risk production issue in the diff.

2. **Harden `startForegroundForAction` against unknown actions** (`CheckInActionService`). Replace the silent `?: return` with a fallback `startForeground` call using a generic string. Add a compile-time exhaustiveness check (e.g. an `else -> R.string.notification_processing_generic` in the `when`) so new action constants do not silently opt out of the foreground promotion.

3. **Expand `RoomWorkEntryRepositoryTest`** to cover at least the normalization path for `upsert`, `replaceEntryWithTravelLegs`, and `readModifyWrite`. The complex recursive locking introduced in this commit deserves a dedicated concurrency test to guard against regressions.
