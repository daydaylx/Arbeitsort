### Positive Observations

1. **`ConfirmationSources` centralisation** — Eliminates 8+ scattered magic string literals (`"NOTIFICATION"`, `"UI"`, `"EDIT_SAVE"`, `"DERIVED_STATE"`, `"WORK_BLOCK"`) spread across use-cases, services, and the ViewModel layer. A single canonical object is far more refactoring-safe.

2. **LRU mutex cache in `RoomWorkEntryRepository`** — Replacing the unbounded `ConcurrentHashMap<LocalDate, Mutex>` with a bounded LRU `LinkedHashMap` (128 entries, reference-counted eviction) closes a low-level memory leak. The reference-counting scheme in `withDateLock`/`trimUnusedDateMutexes` is correct: a mutex is only evicted when `references == 0`, which is guaranteed to be true only after the `finally` block runs post-`withLock`.

3. **Coroutine dispatcher stabilisation** — Wrapping `workEntryRepository.getByDateWithTravel`, `getByDate`, and `deleteByDate` in `withContext(ioDispatcher)` in `EditEntryViewModel`, and offloading `buildOverviewMetrics` to `calculationDispatcher` in `OverviewViewModel`, makes these ViewModels fully testable with `UnconfinedTestDispatcher` without `Dispatchers.setMain` tricks.

4. **`CsvExporter.CsvExportResult` sealed interface** — Replacing `Uri?` with a structured result type makes every error case explicit at the call site. `SettingsViewModel.mapCsvExportError` exhaustively pattern-matches all cases. This is the correct approach and will catch new error variants at compile time if the sealed hierarchy grows.

5. **Performance: glow orbs and blur removed** — `MZAppBackdrop` previously rendered two `Box` composables with `Modifier.blur()`. On pre-Android-12 devices `blur()` falls back to software rendering, degrading performance significantly. The solid-surface redesign eliminates this cost entirely.

6. **Travel leg category preserved through save** — `EditEntrySaveBuilder.build` now uses `leg.category` directly (previously it ran `inferCategory` which silently overwrote user-set categories). `EditEntryTravelSections` already passes `TravelLegCategory.OUTBOUND`, `RETURN`, or `OTHER` explicitly, so round-trip fidelity is now correct.

7. **`statusUi` hoisted in `TodayScreen`** — `resolveTodayStatusUi` was previously called inside `StatusCard` (a private composable), creating invisible coupling between state resolution and UI rendering. Lifting it to `TodayContent` makes it explicit and avoids recomputing it on every `StatusCard` recomposition.

### Top 3 Recommendations

1. **Fix `LinkedHashMap` initial capacity in `RoomWorkEntryRepository`** (effort: 1 line).  
   Change the constructor call to `LinkedHashMap((MAX_CACHED_DATE_MUTEXES / LOAD_FACTOR + 1).toInt(), LOAD_FACTOR, true)` to prevent a silent rehash of up to 96 mutex objects on busy days. This is the only correctness-adjacent concern in this week's changes.

2. **Replace magic scroll-offset arithmetic in `HistoryScreen` with key-based scroll lookup**.  
   The `HISTORY_LIST_CONTENT_START_INDEX + conditional_offset + entryIndex` formula will silently break the next time a sticky header or notice item is added before the entry list. A one-time investment in key-based `LazyListState` navigation will make the scroll logic maintainable long-term.

3. **Consolidate `@VisibleForTesting` dispatcher pattern**.  
   Both `EditEntryViewModel` and `OverviewViewModel` now expose `internal var` dispatcher fields. Before this pattern spreads further, consider introducing a `CoroutineDispatchers` data class injected via Hilt and bound in a test module — a standard Android architecture pattern that removes the mutable field risk while preserving full testability.
