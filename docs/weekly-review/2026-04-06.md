### Positive Observations

- **Thorough test coverage for the audit fixes.** The three new test cases in `AggregateWorkStatsTest` (`averageWorkHoursPerDay exkludiert COMP_TIME-Tage`, `ist 0 wenn nur COMP_TIME-Tage`, `isMealAllowanceEligible ist false fuer FREI_MIT_REISE`) directly pin the previously wrong behaviour and will prevent regression. `ClassifyDayTest` was updated in the same commit to stay consistent.
- **Request-ID guard pattern in `TodayViewModel`.** The introduction of `entryLoadRequestId` / `weekOverviewRequestId` with the `isLatestEntryLoad` / `isLatestWeekOverviewRequest` guards is a clean and lightweight solution to the stale-update race condition. It avoids the complexity of `StateFlow`-based cancellation while still being easy to reason about.
- **Dead-code removal.** `targetHoursForPeriod()` in `OverviewCalculations.kt` was never called by production code. Removing it (and the corresponding test) reduces the surface area for confusion.
- **Named constants over magic numbers.** Extracting `WORKDAYS_PER_WEEK`, `APPROX_WORKDAYS_PER_MONTH`, `WEEKLY_TOLERANCE_HOURS`, and `MONTHLY_TOLERANCE_HOURS` into named file-level constants in `SettingsSections.kt` makes the business rules explicit and searchable.

### Top 3 Recommendations

1. **Fix the `@Volatile` / `AtomicLong` omission in `TodayViewModel` before the next release.** The `observeTodayDate` coroutine runs on `Dispatchers.Default` and writes `weekOverviewRequestId` without memory-visibility guarantees. On most devices this will be invisible because of JVM JIT behaviour, but it is formally a data race and can theoretically manifest as the week overview silently not refreshing after midnight.

2. **Revert the overtime-target tolerance to the previous values (`1.0 h` / `5.0 h`).** The constant extraction was a good refactor, but the values were changed at the same time, which is a hidden behaviour change. Users who set targets like `8.5 h/day` will now see a spurious "inconsistent targets" warning that was never there before.

3. **Harden or remove the string-based exception guard in `closePdfDocumentQuietly`.** Consider whether the double-close scenario can even still occur after the `finally`-block restructuring. If not, remove the wrapper entirely. If it can still occur, suppress the exception unconditionally (and log it at DEBUG level) rather than depending on the message text.
